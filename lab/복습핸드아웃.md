# 복습 핸드아웃: DevSecOps(Infrastructure)

집에 가서 5분이면 오늘 배운 걸 되짚을 수 있는 한 장 정리입니다.

## 오늘 한 줄 요약
> 인프라가 '코드'가 되면, 배포 전에 그 코드를 스캐너로 검사할 수 있다. 위험한 설정을 **왼쪽(개발 단계)에서 미리** 잡는다.

## 핵심 흐름
```
IaC 작성 → 로컬 스캔(tfsec/checkov) → 수정 → 커밋 훅 → CI 스캔 → 머지
```
같은 사이클을 Terraform · 쿠버네티스 · Dockerfile에 똑같이 적용한다.

## 꼭 기억할 규칙
| 영역 | 위험 | 안전 |
|---|---|---|
| S3/스토리지 | public-read, 미암호화 | 비공개 + 암호화 + 퍼블릭 차단 |
| 네트워크(보안그룹) | 22·3306을 `0.0.0.0/0` | 필요한 IP 대역만 |
| 비밀번호·키 | 코드에 하드코딩 | 변수 · 시크릿 매니저 |
| 쿠버네티스 | privileged, root, hostPath | non-root, 최소 권한, resource limits |
| Dockerfile | latest 태그, root 실행 | 버전 고정, 일반 사용자, HEALTHCHECK |

## 오늘 쓴 명령어 (사실상 이게 전부)
```bash
./scan.sh tfsec 01-terraform/vulnerable   # 스캔
diff vulnerable/main.tf fixed/main.tf      # 무엇이 바뀌었나
git commit -m "..."                        # 커밋(pre-commit 훅이 검사)
```

## 자가 점검 (설명할 수 있으면 체크)
- [ ] IaC가 뭔지, 왜 보안에 좋은지 한 문장으로 말할 수 있다.
- [ ] `0.0.0.0/0`이 왜 위험한지 안다.
- [ ] 스캔 결과에서 심각도·규칙ID·위치를 찾을 수 있다.
- [ ] CRITICAL·HIGH부터 고치는 이유를 안다.
- [ ] 비밀번호를 코드에 두면 안 되는 이유를 안다.
- [ ] pre-commit이 왜 "커밋을 막는지" 안다.
